The Pushling is born **per-machine**. One laptop, one creature, one
lifetime. It feeds on all repos worked on from that machine, which is what
makes it feel like a genuine companion — it knows *you*, not just one
project. This concept covers naming, the birth-time computation of
personality and visual traits, and the rarity/shiny system layered on top.
The philosophy behind why identity is *given*, not chosen by Claude, is in
[the vision concept](/vision.md#the-dual-layer-embodiment-model); the
resulting axes are consumed every frame by
[the behavior stack](/SYSTEMS/behavior-stack.md) and described in full in
[personality & emotional state](/REFERENCE/personality-emotional-state.md).

**This concept documents two birth mechanisms that both exist in the
codebase but are not both wired into the live flow — see
[Adjudication: Two Birth Mechanisms](#adjudication-two-birth-mechanisms)
below before treating either as the sole truth.**

# Naming

Deterministic two-syllable names generated by `NameGenerator`
(`Pushling/Sources/Pushling/Creature/NameGenerator.swift`):
`SHA256(git_user.email + machine_hardware_UUID)`, taking the first two hash
bytes modulo the two syllable-table lengths.

| Table | Count | Values |
|---|---|---|
| First syllable | 12 | Pip, Nub, Zep, Tik, Mox, Glo, Rux, Bim, Quo, Fen, Dax, Yol |
| Second syllable | 12 | -o, -i, -us, -el, -a, -ix, -on, -y, -er, -um, -is, -ot |

12 × 12 = 144 possible names (e.g. Pipo, Zepus, Moxa, Ruxon, Fenum, Daxis).
Same email + same machine always yields the same name — deterministic, not
random. The machine UUID is read via `sysctl -n kern.uuid`, falling back to
`ioreg -rd1 -c IOPlatformExpertDevice`'s `IOPlatformUUID` if that's empty,
and to the literal string `"unknown-machine"` if both fail. The email
defaults to `"unknown@pushling"` if `git config --global user.email` is
unset. The creature is renameable via MCP at any time after birth; renaming
does not change any other trait.

# The Live Birth Flow: Progressive Learning via EggAccumulator

**What actually runs today.** At first launch with no existing creature
row, `GameCoordinator.wireHatching()` starts the 30-second
`HatchingCeremony` (montage → materialization → naming — see
[Stage Transitions](/REFERENCE/growth-stages.md#stage-transition-ceremony)
for the shared ceremony choreography) and, in parallel, instantiates an
`EggAccumulator` (`Pushling/Sources/Pushling/Creature/EggAccumulator.swift`).
The ceremony completes with a **neutral** personality and neutral visual
traits — `completeHatching()` sets `creatureStage = .egg`, `totalXP = 0`,
and explicitly comments "real personality computed later from commits via
EggAccumulator." The creature emerges from the egg looking and acting
generic; its identity has not been decided yet.

From that point, every commit the creature eats while at `egg` stage is
recorded into the accumulator (`EggAccumulator.record(_:)`): message
length, files changed, lines added/removed, repo name, timestamp, and
per-language counts. Once **5 commits** have been absorbed
(`EggAccumulator.hatchThreshold = 5`), `GameCoordinator.hatchEggIntoDrop()`
fires:

1. **Identity seed.** `EggAccumulator.gitEmailHash()` computes an FNV-1a
   hash of `git config user.email` (falls back to `"unknown"`) — a `UInt32`
   seed that is deterministic per-developer.
2. **Rarity + shiny draw.** `EggAccumulator.determineRarity(seed:)` runs a
   Mulberry32 seeded PRNG twice from that seed: draw 1 picks a
   [`RarityTier`](#rarity-tiers), draw 2 is an independent 1% shiny check.
   Same email → same rarity → same shiny, every time, on every machine.
3. **Identity biases.** `EggAccumulator.computeIdentityBiases(seed:)` draws
   5 more Mulberry32 values from an *offset* seed (`seed ^ 0xDEADBEEF`, kept
   independent of the rarity draws) producing: a hue shift (±0.08), an eye-
   shape tendency, a tail-shape tie-break tendency, and four personality-axis
   biases (±0.05 each, one per axis below). These are transient — computed
   once, baked into the persisted `Personality`/`VisualTraits`, never stored
   separately.
4. **Personality from commit data.** `EggAccumulator.computePersonality(
   raritySpread:identityBias:)` computes each axis from the 5 accumulated
   commits (formulas below), applies the identity bias additively, then
   applies the rarity's `spreadBonus` multiplicatively around the neutral
   midpoint (`0.5 + (axis - 0.5) * (1.0 + spreadBonus)`) — bias shifts the
   center, rarity spread amplifies how far from center it lands.
5. **Visual traits from commit data**, similarly bias-adjusted (hue, eye
   shape, tail shape tie-break).
6. Stage advances from `egg` to `drop`; the accumulator is discarded
   (`eggAccumulator = nil`); skill stats initialize from the same egg data
   (`initializeSkillStatsFromEgg()`); rarity visual effects apply
   (`CreatureNode.setupRarityEffects`).

**Personality formulas actually used at this step** (distinct from the
formulas in [GitHistoryScanner](#the-designed-but-unwired-flow-githistoryscanner),
below — the two compute the same four axes differently):

| Axis | Formula (from the 5-commit sample) |
|---|---|
| Energy | `1.0 - (avg_commit_interval_seconds / 1800)`, clamped [0,1] — short gaps between the 5 commits (bursty) read as high energy |
| Verbosity | `(avg_message_length - 10) / 60`, clamped [0,1] |
| Focus | `1.0 - (avg_files_per_commit - 1) / 10`, clamped [0,1] — fewer files per commit reads as more focused |
| Discipline | fixed `0.5` (neutral) — the comment in code notes this is "limited with so few data points" and is not actually computed from timing regularity at this stage |
| Specialty | the single most-committed `LanguageCategory` across the 5 commits, or `.polyglot` if none matched |

# Rarity Tiers

`RarityTier` (`Pushling/Sources/Pushling/Creature/PersonalitySystem.swift`)
is a from-birth roll baked permanently into the creature — not something
that changes with play:

| Tier | Chance | `spreadBonus` (personality) | `statFloor` (skill stats) |
|---|---|---|---|
| Common | 60% | +0% | 10 |
| Uncommon | 25% | +5% | 15 |
| Rare | 10% | +10% | 20 |
| Epic | 4% | +15% | 25 |
| Legendary | 1% | +20% | 35 |

Cumulative-distribution roll: `common ≤ 0.60`, `uncommon ≤ 0.85`,
`rare ≤ 0.95`, `epic ≤ 0.99`, `legendary ≤ 1.00`. An independent 1% "shiny"
flag can co-occur with any tier. Both `rarity` (TEXT, CHECK'd against the
five tier names) and `shiny` (INTEGER 0/1) are SQLite columns added by
schema migration v5 (`Pushling/Sources/Pushling/State/Migration.swift`), not
present in the base `creature` table definition. `statFloor` guarantees
rarer creatures never decay a skill stat below a permanent baseline — see
`Pushling/Sources/Pushling/Creature/SkillStats.swift`.

# Visual Traits

Computed the same way as personality — from the 5-commit `EggAccumulator`
sample, bias-adjusted, not from the full historical scan:

| Trait | Derived From | Notes |
|---|---|---|
| `baseColorHue` | `specialty.baseColorHue` + identity `hueShift`, wrapped to [0,1) | See the hue table in [personality & emotional state](/REFERENCE/personality-emotional-state.md#language-specialty-categories) |
| `bodyProportion` | equals the computed `focus` personality axis | Not an independent add/delete-ratio calculation in the live `EggAccumulator` path (contrast with `GitHistoryScanner`, below, which does compute it from lines added/deleted) |
| `furPattern` | `FurPattern.fromRepoCount()` — none (1-3 repos), spots (4-8), stripes (9-15), tabby (16+) | Repo count = number of distinct repos among the accumulated commits |
| `tailShape` | `TailShape.fromCategory(specialty)` — thin (systems), fluffy (frontend/backend), serpentine (script), else `.standard` | When the category yields `.standard` (no strong signal), the identity `tailTendency` bias breaks the tie: [0, 0.25) thin, [0.25, 0.5) serpentine, [0.5, 0.75) fluffy, [0.75, 1] stays standard |
| `eyeShape` | `EyeShape.fromAverageMessageLength()` — round (avg > 50 chars), narrow (avg < 20), else standard | Identity `eyeTendency` bias shifts the effective average by ±15 chars before this lookup |

# The Designed-but-Unwired Flow: GitHistoryScanner

`GitHistoryScanner` (`Pushling/Sources/Pushling/Creature/GitHistoryScanner.swift`)
is a fully-implemented, 30-second-timeout, background-thread scanner that
walks `~/Projects`, `~/Developer`, `~/code`, `~/github`, `~/repos`, `~/src`,
`~/work` (depth ≤ 4, skipping `node_modules`/`.git`/build dirs/etc.), runs
`git log --author=<email> --numstat --no-merges -n 5000` per repo, and
computes the **entire historical-lifetime** personality and visual-trait
set — a materially richer computation than the 5-commit `EggAccumulator`
path:

| Axis | `GitHistoryScanner` formula (lifetime data) |
|---|---|
| Energy | `burst_ratio × 0.7 + commits_per_day_normalized × 0.3` — `burst_ratio` is the fraction of commits inside 5+-commit bursts within a 1-hour window |
| Verbosity | `avg_message_length / 100`, clamped [0,1] |
| Focus | `1.0 - files_factor × 0.5 - repo_switch_frequency × 0.5` |
| Discipline | `1.0 - stddev(commit_hour) / 6.0` — **actually computed** from timing regularity, unlike the egg-accumulator path's fixed 0.5 |
| Specialty | dominant `LanguageCategory` by extension count across every repo, or `.polyglot` if none exceeds 30% |

Visual traits additionally compute `bodyProportion` from the true lifetime
add/delete ratio (net-adder ≥1.5 → 0.7 rounder; net-deleter ≤0.7 → 0.3
leaner; else 0.5) — the calculation the vision doc describes under "Body
proportions." `HatchingCeremony.feedScanResult(_:)` exists to receive a
`GitScanResult` and populate the 20-second montage (repo names, language
badges, commit/line counts scrolling past) plus set the ceremony's
`personality`/`visualTraits` from it, and `feedEmptyResult()` exists as the
graceful "no commits yet" flavor for a fresh git identity.

**Neither `GitHistoryScanner.scan()` nor `feedScanResult(_:)` nor
`feedEmptyResult()` is called from anywhere in the running app** (verified:
zero call sites for `GitHistoryScanner.scan(` outside its own file; zero
call sites for `feedScanResult`/`feedEmptyResult` outside
`HatchingCeremony.swift` itself). `GameCoordinator+Hatching.swift` explicitly
comments "No git scanner — personality will be learned progressively from
commits during the egg stage via EggAccumulator" at the exact point where a
scan would have been kicked off. The practical effect: the 20-second montage
phase runs with `scanResult == nil` for its full duration, so
`spawnNextMontageItem()` falls into its no-data branch and shows a
`"scanning..."` placeholder with cycling dots the entire time — the "rapid
montage of git history scrolling past" the vision doc and this ceremony's
own code comments describe never actually appears.

# Adjudication: Two Birth Mechanisms

This is a **new drift signal**, not one flagged by the original survey —
`GitHistoryScanner` is real, complete, tested-looking code, not a stub, and
was not caught because nothing in the source docs pointed a reviewer at "is
this scanner actually invoked." Per DOCS WIN, the concept must state what
happens today (the `EggAccumulator` progressive-learning flow, which is
real and running) as canon, while treating `GitHistoryScanner`'s richer
lifetime computation and the montage experience as **designed, implemented,
currently disconnected** — this is a lens-3 (defined-but-unwired) gap, not
stale intent to prune and not yet-canon-because-unbuilt (it *is* built).
Flagged for `DECISIONS.md`/the Orchestrator: should `GitHistoryScanner` be
wired back into `wireHatching()` (restoring the vision doc's "history
shapes appearance and personality from birth, XP still starts at 0" model
verbatim), or should the vision doc's Historical Seeding section be revised
to describe the `EggAccumulator` progressive-learning design as the
intended replacement? Both readings are defensible; this concept does not
adjudicate it and instead documents both mechanisms precisely so the
decision can be made with full information.

The vision doc's own **Historical Seeding** framing —
"a developer with 10,000 PHP commits gets a purple-hued cat spirit with
web-specialist traits... but the creature still starts at 0 XP with 0 eaten
commits, history determines who it *is*, not how far along it is" — is
preserved below as superseded design history, since it describes the
`GitHistoryScanner` path that is not currently reachable:

> On first install, the daemon scans all repos and counts historical
> commits. This history shapes the creature's appearance and personality
> from birth... During the ~30-second "hatching ceremony," the user sees a
> rapid montage of their git history scrolling past — repo names, language
> badges, commit counts.

# Adjudication: `specialty` Column CHECK Constraint Mismatch

A second **new drift signal**, found while verifying persistence: the base
`creature` table's `specialty` column CHECK constraint
(`Pushling/Sources/Pushling/State/Schema.swift`) allows exactly
`'polyglot','frontend','backend','systems','data','mobile','devops',
'scripting','functional','creative','research'` — but `LanguageCategory`
(the enum both `Personality.specialty` and `VisualTraits` actually store,
matching the vision doc's 11-category table) has raw values `systems,
frontend, backend, script, jvm, mobile, data, infra, docs, config,
polyglot`. Six of eleven values differ (`script`≠`scripting`, `jvm` absent,
`infra`≠`devops`, `docs` absent, `config` absent — with `devops, scripting,
functional, creative, research` in the CHECK constraint corresponding to no
enum case at all). `PersonalityPersistence.save()` wraps the `UPDATE` in a
`do/catch` that only `NSLog`s on failure, so any creature whose specialty
resolves to `.script`, `.jvm`, `.infra`, `.docs`, or `.config` will silently
fail to persist its specialty (and, since it's one `UPDATE` statement with
multiple columns, this failure mode needs verifying against whether it
aborts the whole statement or just that CHECK) — flagged for
`DECISIONS.md`/the Orchestrator as a genuine code bug to fix (code should
conform to the `LanguageCategory` enum, which matches vision-doc canon; the
CHECK constraint is what's wrong here), not a documentation question.
