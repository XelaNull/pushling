Four scripts, one shared version file (`.build-version`), one build product
(`build/Pushling.app`). This concept documents what each script actually
does, including two behaviors worth knowing before you rely on them (a
double-launch case in `run.sh`, and when `HotReloadMonitor`'s self-restart
path is and isn't actually exercised).

# build.sh ŌĆö Build, Sign, Deploy

```bash
./build.sh              # Debug build, auto-increment build number
./build.sh release      # Release (optimized) build
./build.sh patch        # Bump patch, reset build to 0, then increment to 1
./build.sh minor        # Bump minor, reset patch+build
./build.sh major        # Bump major, reset minor+patch+build
```

**Versioning** (`MAJOR.MINOR.PATCH.BUILD`, stored in `.build-version` at the
repo root): the build component auto-increments on *every* invocation
regardless of arguments; `patch`/`minor`/`major` additionally bump their
semver component and reset everything below it to 0 before the build bump
applies. `Pushling/Sources/Pushling/App/PushlingVersion.swift` is
regenerated from this value on every `build.sh` run (a literal
`PushlingVersion` enum with `major`/`minor`/`patch`/`build`/`string`) ŌĆö it
carries an explicit `// AUTO-GENERATED by build.sh ŌĆö do not edit manually`
header and is overwritten unconditionally, so any manual edit is lost on
the next build.

**Build steps:**
1. `swift package clean` (unconditional ŌĆö works around an SPM incremental
   cache bug where stale `.o` files get served when only function bodies
   change) then `swift build` with `-c debug` or `-c release`.
2. Assemble `build/Pushling.app` from scratch (`rm -rf` then `mkdir -p` the
   bundle structure) ŌĆö no incremental bundle updates.
3. Generate `Info.plist`: `CFBundleIdentifier` = `com.pushling.app`,
   `LSUIElement = true` (no Dock icon, no app-switcher entry),
   `LSMinimumSystemVersion = 14.0`, version strings from the semver/build
   computed above, plus the current short git hash and UTC build date.
4. Copy the built binary and any `Resources/*.{png,jpg,wav,mp3,json}` +
   `com.pushling.daemon.plist` into the bundle.
5. **Ad-hoc code sign:** `codesign --force --sign - --deep`. Not signed
   with a Developer ID ŌĆö fine for local development, but the resulting
   bundle will trigger Gatekeeper on another machine.
6. **Deploy:** if `/Applications/Pushling.app` already exists (true after
   `install.sh` has run once, and it stays true across subsequent
   `build.sh` runs), `build.sh` **itself** stops any running `Pushling`
   process (`killall`, waits up to 3s, then `killall -9` if still alive),
   `rsync -a --delete`s the fresh bundle into `/Applications/`, and
   `open`s it. If `/Applications/Pushling.app` does not yet exist, it's
   installed fresh (`cp -R`) and launched the same way. Either branch means
   **`build.sh` deploys and relaunches on its own** ŌĆö it does not wait for
   or rely on `HotReloadMonitor` to notice the new binary (see below).

# run.sh ŌĆö Build (optional) + Launch

```bash
./run.sh              # Debug build + launch
./run.sh release       # Release build + launch
./run.sh --no-build    # Launch the existing build/Pushling.app without rebuilding
```

Sequence: kill any running `Pushling` process (`pkill -x`), optionally call
`build.sh "${CONFIG}"`, then unconditionally `open build/Pushling.app`.

**Known double-launch case:** if `/Applications/Pushling.app` exists (the
common case in a dev environment where `install.sh` has run) and `run.sh`
is invoked *without* `--no-build`, two Pushling processes end up running:
`build.sh`'s own deploy step (step 6 above) already opens
`/Applications/Pushling.app`, and then `run.sh` separately opens
`build/Pushling.app` right after `build.sh` returns ŌĆö two different bundle
paths, both launched. This is a real behavior of the scripts as written
today (verified by tracing both scripts' control flow), not a hypothetical
ŌĆö flagged here rather than silently smoothed over, since a developer
debugging "why are there two Pushling processes / two creatures on the
Touch Bar" needs this explained. `./run.sh --no-build` (launch only, no
rebuild) does not hit this, since it skips `build.sh` entirely.

# reload.sh ŌĆö Build and Let Hot-Reload Handle Restart

```bash
./reload.sh              # Debug build
./reload.sh release      # Release build
```

Calls `build.sh "${1:-debug}"`, then checks whether a `Pushling.app`
process is running and prints a message expecting `HotReloadMonitor` to
detect the new binary and gracefully self-restart.

**In practice, this expectation is not what happens when
`/Applications/Pushling.app` already exists:** `build.sh`'s own deploy step
(above) explicitly kills the running process *before* `HotReloadMonitor`
in that (still-alive, for another moment) process would ever get a chance
to detect the binary swap ŌĆö the old process is torn down, then the new
bundle is rsync'd in, then a fresh process is launched directly via `open`.
`HotReloadMonitor`'s directory-watch mechanism (see
[persistence and recovery](/OPERATIONS/persistence-and-recovery.md) for its
mechanics) is real and correctly implemented, but none of the four scripts
in this repo currently exercise the path it's built for ŌĆö an in-place
binary swap **underneath a still-running process that nobody explicitly
stopped first**. It functions as a safety net for that scenario (e.g. an
out-of-band deploy tool, or a manual `rsync` into a live install) rather
than as the mechanism these day-to-day scripts rely on. `reload.sh`'s
printed fallback ŌĆö manually POST `{"command":"reload"}` to the socket if
nothing restarts within 5 seconds ŌĆö remains valid as a way to force the
same graceful-restart code path on demand.

# install.sh ŌĆö Install as a Login Item

```bash
./install.sh              # Release build, install, register LaunchAgent
./install.sh --uninstall  # Stop, unload LaunchAgent, remove from /Applications
```

Always builds `release` (never debug). Stops any running instance, unloads
any existing LaunchAgent plist, copies the fresh bundle to
`/Applications/Pushling.app`, writes
`~/Library/LaunchAgents/com.pushling.daemon.plist`, and `launchctl load`s
it. The plist:

| Key | Value |
|---|---|
| `Label` | `com.pushling.daemon` |
| `ProgramArguments` | `["/Applications/Pushling.app/Contents/MacOS/Pushling"]` |
| `RunAtLoad` | `true` |
| `KeepAlive` | `true` (auto-restart on crash ŌĆö the basis for the heartbeat/crash-recovery contract) |
| `ProcessType` | `Interactive` (higher scheduling priority, appropriate for a 60fps UI process) |
| `StandardOutPath` / `StandardErrorPath` | `~/Library/Logs/Pushling/pushling.{stdout,stderr}.log` |
| `EnvironmentVariables` | `PUSHLING_LAUNCHED_BY_AGENT=1` |

`--uninstall` preserves all state data (`~/.local/share/pushling/`) ŌĆö only
the app bundle and LaunchAgent registration are removed; the script prints
the manual `rm -rf` command for a full data wipe rather than doing it
itself.

# In-App LaunchAgent Toggle (undocumented-elsewhere path)

Beyond `install.sh`, the running app itself exposes a menu item ("Install
Launch Agent" / "Remove Launch Agent", toggled via
`AppDelegate.toggleLaunchAgent`) backed by a separate
`LaunchAgentManager` class that writes the **same**
`com.pushling.daemon.plist` at the same path and label ŌĆö but with
`ProgramArguments` pointed at
`ProcessInfo.processInfo.arguments[0]`, i.e. **whatever binary is currently
running**, not necessarily `/Applications/Pushling.app`. If toggled while
running the `build/Pushling.app` dev copy (e.g. via `run.sh`), the
resulting LaunchAgent will auto-relaunch that dev path on login, not the
`/Applications` copy. This is a legitimate, working, code-verified
alternate path to `install.sh` for auto-start ŌĆö not previously documented
in `pushling/CLAUDE.md`'s Essential Commands section ŌĆö flagged here as new
information for the Orchestrator/`DECISIONS.md` rather than assumed to be a
bug, since both paths converge on the same LaunchAgent contract and either
one is a valid way to get an auto-starting instance.

# --test-mode ŌĆö Isolated Accelerated Lifecycle Simulator

```bash
build/Pushling.app/Contents/MacOS/Pushling --test-mode \
  [--xp-multiplier=100] [--time-accel=1440] [--auto-feed] \
  [--commits-per-day=10] [--workbench-scale=6]
```

An alternate launch path (checked in `AppDelegate.applicationDidFinishLaunching`
before the workbench check, and before the normal daemon path) that runs a
"spore-to-apex" lifecycle simulation instead of either of those ŌĆö built to
QA growth-stage visuals/transitions/hatching across a full creature life in
minutes instead of months, without touching the developer's real creature.
Composes with `main.swift`'s single-instance guard the same way `--workbench`
does: it's exempted, so it can run alongside a real daemon.

**Flags** (all optional; parsed by `TestModeConfig.fromProcessArguments()`
in `TestMode.swift`):

| Flag | Default | Effect |
|---|---|---|
| `--test-mode` | ŌĆö | Required to activate; every other flag below is a no-op without it. |
| `--xp-multiplier=N` | `100` | Multiplies the XP every synthetic commit awards (same formula real commits use ŌĆö see below). |
| `--time-accel=N` | `1440` | `TestTimeProvider`'s simulated-time acceleration factor (1440x = 1 real minute per simulated day). Note: `LifecycleSimulator`'s own day-counting uses a fixed 1-real-minute-per-day clock independent of this value ŌĆö `time-accel` currently only affects `TestTimeProvider.now`, not day-advancement pacing. |
| `--auto-feed` | off | Parsed into `TestModeConfig.autoFeed` but not yet read by the simulation loop ŌĆö reserved for a future batch-commit-file mode; has no effect today. |
| `--commits-per-day=N` | `10` | Parsed into `TestModeConfig.simulatedCommitsPerDay` but not yet read by `LifecycleSimulator` (which computes its own daily target internally ŌĆö see `LifecycleSimulator.startNewDay`); has no effect today. |
| `--workbench-scale=N` | `6` | Same flag `--workbench` reads (`WorkbenchMode.magnification`) ŌĆö test-mode always presents through the magnified workbench window (see below), so this sizes it same as it would under plain `--workbench`. |

**Isolation (inert against real state):** `AppDelegate.setupTestMode()`
starts its own `StateCoordinator` against a scratch database path under
`NSTemporaryDirectory()` (`pushling-test-mode-<pid>.db`) ŌĆö never the real
`~/.local/share/pushling/state.db` ŌĆö with `persistenceEnabled: false`, the
exact pattern `--workbench` uses (see `WorkbenchMode.swift` /
`AppDelegate.setupWorkbench`). Every `GameCoordinator` DB write
(`persistXPAndStage`, `checkEvolution`, journal inserts) goes through
`stateCoordinator.database`, which is bound to that scratch path ŌĆö that's
what keeps a simulated life off the real creature. `persistenceEnabled:
false` additionally skips the heartbeat writer and the backup check, which
sit at fixed absolute paths (`/tmp/pushling.heartbeat`,
`~/.local/share/pushling/backups/`) independent of the database path.

`LifecycleSimulator`'s synthetic commit JSON (`onSimulatedCommit`) is fed
directly into that isolated `GameCoordinator`'s
`feedProcessor.onCommitReceived` ŌĆö the same entry point real git commits
reach via the feed directory ŌĆö so XP, stage transitions, hatching, and
mutation badges run through the real game logic rather than a parallel
reimplementation. The feed directory's own file-watcher is never started
(gated behind `persistenceEnabled` inside `GameCoordinator.init`,
unchanged), so the real `~/.local/share/pushling/feed/` directory is never
read either. `LifecycleSimulator` stops itself and prints a final report
via `NSLog` once the simulated creature reaches `.apex`.

**A pre-existing gap this closed:** `StateCoordinator.start()` used to
construct a throwaway `HeartbeatManager()` for the pre-database-open crash
check, then replace it with a second instance "now that the DB reference is
available." That replacement's implicit deinit called
`HeartbeatManager.stop()` -> `writeShutdownMarker()` ŌĆö which writes to the
fixed `/tmp/pushling.heartbeat` path **unconditionally, with no
`persistenceEnabled` guard**. Every `persistenceEnabled: false` launch
(previously only `--workbench`, now also `--test-mode`) was silently
overwriting the real daemon's heartbeat file the instant `start()` ran,
contradicting `WorkbenchMode.swift`'s own documented invariant. Fixed by
constructing exactly one `HeartbeatManager` for the whole call (passing the
`DatabaseManager.shared` reference up front ŌĆö valid even before `.open()`,
since `HeartbeatManager` already guards every DB-touching path on
`db.isOpen`) so it's never prematurely replaced/deinit'd. This benefits
`--workbench` too, not just `--test-mode`.

**Composing with `--workbench`:** `--test-mode` always presents through
`WorkbenchWindowController` internally (never the physical Touch Bar, so a
test-mode run never fights a real running daemon over the private Touch Bar
API). Passing `--workbench` alongside `--test-mode` is therefore a no-op ŌĆö
identical to `--test-mode` alone ŌĆö rather than a conflict or a
silently-dropped flag. `--workbench` on its own (no `--test-mode`) is
unaffected: it still shares the real daemon's `state.db` read/animate-only,
per its own section above.

# bin/pushling ŌĆö Standalone Operator CLI

A fully-implemented (not a stub) bash CLI exists at `bin/pushling`
(~660 lines) ŌĆö verified during this wave, and worth stating plainly since
`PUSHLING_VISION.md`'s Installation section describes similar subcommands
in a context that reads as partly aspirational (see the accompanying
traceability record's correction). Its subcommands, dispatched from a
single `case` statement:

| Command | What it does |
|---|---|
| `pushling track [path]` / `untrack [path]` | Installs/removes the canonical `hooks/post-commit.sh` as `.git/hooks/post-commit` in a repo, backing up any pre-existing hook to `post-commit.pre-pushling`; signals the daemon over the socket (`track_repo`/`untrack_repo`) if it's running |
| `pushling hooks install` / `remove` / `status` | Generates 7 thin Claude Code hook scripts under `~/.pushling/hooks/` (one per `CLAUDE_HOOK_TYPES` entry) that write feed JSON and signal the daemon, registers them via `claude hooks add`, and registers the MCP server via `claude mcp add pushling -- node ~/.pushling/mcp/dist/index.js` |
| `pushling export [file]` / `import <file>` | Exports the `creature` row + last 500 `journal` entries to a portable JSON file (daemon-signaled first, direct-SQLite-via-`sqlite3`+`python3` as fallback); import rebuilds `state.db` from a JSON file after backing up the existing one, with a confirmation prompt |
| `pushling voice download [--tier1-only\|--tier2]` / `voice status` | Delegates to `bin/pushling-voice-setup` (a separate, also fully-implemented ~500-line script that downloads the 3 sherpa-onnx TTS model tiers from pinned `k2-fsa/sherpa-onnx` GitHub releases) |
| `pushling status` | Daemon running? (socket file exists), heartbeat file mtime, database presence + creature name/stage/commits/XP via `sqlite3`, pending feed-file count |
| `pushling logs` | Tails `~/Library/Logs/Pushling/pushling.log`, falling back to `log show --predicate 'process=="Pushling"'` |
| `pushling restart` | `launchctl unload`/`load` if a LaunchAgent plist exists, else `killall`+`open` the `/Applications` copy |
| `pushling uninstall [--purge]` | Stops the daemon, removes hooks from every tracked repo (queried from the `repos` table), removes Claude Code hooks + MCP registration, removes the app; `--purge` additionally deletes `~/.local/share/pushling/` |

This script is **not currently invoked by, or wired into,
`build.sh`/`install.sh`** ŌĆö none of those four scripts copy `bin/pushling`
into the built `.app` bundle or onto a user's `$PATH`. It exists as
ready-to-use source but has no install path yet other than running it
directly from a repo checkout (`./bin/pushling <command>`).

# Formula/pushling.rb ŌĆö Homebrew Cask (Not Yet Publishable)

A real Homebrew cask formula exists at `Formula/pushling.rb`, matching
`PUSHLING_VISION.md`'s `brew install --cask pushling` Installation section
closely (same subcommand names, same data directories). It is **not
publishable as-is**: `sha256 "TO_BE_COMPUTED"` is a literal placeholder, and
its `url` points at
`https://github.com/pushling/pushling/releases/download/...` ŌĆö no such
release, and no evidence this repo publishes to that GitHub organization.
It also expects `#{appdir}/Pushling.app/Contents/Resources/bin/pushling`
and a `Contents/Resources/mcp` directory to exist inside the built app
bundle for its `binary`/`artifact` stanzas to succeed ŌĆö `build.sh`'s
Resources-copying step (above) does neither today, so even with a real
release URL and computed checksum, installing this cask against the
current `build.sh` output would fail at the `binary`/`artifact` steps.

Per Ruling R4, this aspirational-but-partially-real distribution story is
preserved rather than dropped ŌĆö but it belongs in a `FEATURES/`-type
concept (where `­¤ōÉ` status markers are permitted) documenting "Homebrew
distribution" as a designed-but-not-yet-wired feature, not in this
Runbook, which only documents what a developer can reliably run *today*.
Flagged for the Orchestrator/a later wave (no `FEATURES/` concept for
distribution exists yet in this bundle as of this wave).
